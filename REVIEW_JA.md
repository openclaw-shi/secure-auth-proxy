# コードレビュー（日本語）

対象: `src/server.ts`

## 総評

- 構成はシンプルで、設定ファイル駆動のサービス切り替え、ヘッダサニタイズ、UDS/TCP 切替が明確に実装されています。
- 一方で、運用時の安定性とセキュリティの観点で、優先的に改善したい点がいくつかあります。

## 指摘事項

### 1. Upstream リクエストにタイムアウトがない（重要）

`fetch` 呼び出しにタイムアウトがなく、上流 API のハング時に接続が長時間ぶら下がる可能性があります。

**提案**
- `AbortController` を使い、サービスごとにタイムアウト（例: 30 秒）を設定する。
- タイムアウト時は 504 を返すなど、502 と区別すると障害切り分けしやすい。

### 2. `serviceName` を正規表現へ直接埋め込み（中）

`new RegExp(
  \\`^/${serviceName}\\`
)` のように、サービス名をエスケープせず正規表現へ埋め込んでいます。現在の設定値次第では問題化しませんが、将来的にサービス名に正規表現メタ文字が含まれると意図しないマッチが起こり得ます。

**提案**
- 正規表現ではなく、文字列スライスで prefix を除去する。
- もしくは正規表現エスケープ関数を通してから利用する。

### 3. `DELETE` のボディを常に破棄（中）

`hasBody` 判定で `DELETE` を body なし扱いにしています。実運用では `DELETE` に JSON body を求める API も存在するため、互換性問題になる可能性があります。

**提案**
- `GET` / `HEAD` のみ body なし扱いにする。

### 4. 起動時設定のバリデーション不足（中）

`config.yaml` と `keys.json` の読み込みはしていますが、型・必須項目の厳密検証はありません。設定ミス時に実行中まで潜在化し、初回リクエストで 500 になる可能性があります。

**提案**
- 起動時に `services[*].target`, `auth_header`, `key_id` などを検証して fail-fast にする。
- 可能なら `zod` 等でスキーマ検証を導入する。

## 良い点

- ホップバイホップヘッダ除去が実装され、プロキシとしての基本安全性を担保できています。
- エラー JSON の末尾改行やタイムスタンプログなど、運用観点の配慮がされています。
- UDS と TCP フォールバックの両対応で、デプロイとデバッグの使い分けがしやすいです。

## 推奨アクション（優先順）

1. Upstream タイムアウト実装（最優先）
2. `serviceName` の prefix 除去ロジックを正規表現非依存へ変更
3. `DELETE` body の取り扱い見直し
4. 設定ファイルの起動時バリデーション追加
